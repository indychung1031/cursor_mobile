## 1. 한 줄 정의

> **PC에서 실행 중인 Cursor Agent 채팅 패널을 모바일 브라우저에서 보고 입력할 수 있는 개인용 원격 브리지**

### 전제 조건

#### Phase 0 ~ Phase 1 (현재 착수 범위)

아래 조건을 **항상 만족한다고 가정**하고 개발·테스트한다. 위반 시 B모드는 동작하지 않는다.

| 전제 | 내용 |
|------|------|
| PC 상시 가동 | PC가 켜져 있고 인터넷·Tailscale에 연결되어 있다 |
| **Cursor 전면 실행** | Cursor IDE가 **열려 있고**, Agent 채팅 패널이 **보이는 상태** |
| **화면 잠금 없음** | **Win+L 잠금 화면 사용 안 함** — Phase 0~1에서 검증·지원하지 않음 |
| **시스템 절전 없음** | S3/S4/S5 절전 진입 안 함 |
| **디스플레이 ON** | OS가 디스플레이를 끄지 않음 (Bridge `powercfg` 적용). Electron(Cursor) 정상 렌더링 |
| 단일 사용자 | 본인만 사용하는 개인용 |
| (권장) EDID Emulator | 더미 HDMI/DP 플러그 사용 시 **실제 모니터 전원은 꺼도** OS는 디스플레이 연결 유지 → 창 위치·캡처 안정, 화면 노출·번인 방지 |

> **사용자 결정 (2026-06-27):** Phase 1까지는 위 전제만 다룬다. **화면 잠금 상태 지원은 Phase 3 백로그**로 미룬다.

#### Phase 3+ (향후 — 미착수)

| 항목 | 방향 (개략) |
|------|-------------|
| Win+L 잠금 중 사용 | GUI(B모드) 불가 → CLI/API·RDP·Background Agent 등 **별도 아키텍처** 검토 |
| 시스템 절전 | Wake-on-LAN + 절전 정책 또는 항상 켜 둠 |

## 2. 두 가지 모드

### B모드 — 화면 스트리밍 (1차 개발)

Cursor 채팅 패널 화면을 영상으로 캡처해 모바일에 스트리밍한다.

```
[Cursor Agent 채팅 패널]
        ↓ 화면 캡처 (채팅 패널 영역)
[Bridge Server @ PC]  ←  모바일 텍스트 입력 수신
        ↓ 클립보드 주입 → Cursor 채팅창 포커스 → 붙여넣기
[Tailscale 터널]
        ↓
[모바일 브라우저]
  · 상단: 채팅 패널 영상
  · 하단: 텍스트 입력창 + 전송 버튼
```

**장점**
- Cursor 내부 구조에 의존하지 않아 업데이트에 강함
- 구현 단순 (화면 캡처 + 클립보드)
- **다중 모니터 지원**: 모니터 여러 대 환경에서 원하는 화면 선택 및 전환 가능
- Bridge 실행 중 디스플레이 sleep 자동 방지 → 대기 상태에서도 캡처 유지

**단점**
- 영상이라 텍스트가 약간 흐릿할 수 있음 (Phase 0에서 가독성 먼저 검증)
- 스트리밍 지연 존재 (약 0.5~2초) — 입력 후 화면 반영이 늦어 오타 수정이 어렵고 답답한 UX 유발 가능
- 입력창 고정 좌표 — Cursor가 diff/승인 다이얼로그를 렌더링하면 위치가 이동할 수 있음 (A모드 대비 약점)
- **클립보드 충돌**: 텍스트 전송 시 PC 클립보드를 덮어쓰므로 멀티태스킹 중 기존 복사 내용이 사라짐
- **포커스 타이밍 위험**: 전송 순간 PC에서 다른 창(팝업 등)이 활성화되면 엉뚱한 위치에 텍스트가 입력되거나 의도치 않은 명령이 실행될 수 있음
- **MJPEG 대역폭 소모**: 프레임마다 전체 이미지를 전송하므로 H.264 대비 데이터 사용량이 큼 — LTE 장시간 사용 시 주의

---

### A모드 — Extension + Bridge (향후 고품질 버전)

Cursor 플러그인이 Agent 채팅 내용을 텍스트로 추출해 모바일에 전달한다.

```
[Cursor Extension]
  · 대화 내용 텍스트 추출
  · 메시지 전송 트리거
        ↕ localhost
[Bridge Server @ PC]
        ↕ Tailscale
[모바일 PWA]
  · 말풍선 채팅 UI
  · 텍스트 입력 → Bridge → Extension → Cursor
```

**장점**
- 깔끔한 채팅 UI (말풍선)
- 영상 없이 텍스트만 전송 → 빠르고 선명

**단점**
- **Proprietary API 의존**: Cursor AI 채팅 패널은 VS Code Extension 표준 API 범위 밖의 독자 구현 — 채팅 텍스트 추출이 불가능하거나, 내부 DOM을 직접 접근해야 할 경우 Cursor 패치마다 연동이 깨지는 유지보수 부담 발생. 스파이크 결과에 따라 구현 전체가 취소될 수 있음
- 구현 복잡 (Extension + Bridge + PWA 3계층)

## 4. 개발 로드맵

### Phase 0 — 인프라 구성 + 조기 검증 (코드 최소, 1~2일)

| 작업 | 내용 | 예상 |
|------|------|------|
| Tailscale 설치 | PC + 모바일 같은 계정 등록 | 30분 |
| Bridge 서버 뼈대 | Node.js, health check, 모니터 목록 API | 4시간 |
| **[조기 검증] 캡처 가독성** | 채팅 패널 JPEG 캡처 → 실제 폰으로 텍스트 읽히는지 확인 | 1시간 |
| **[조기 검증] 디스플레이 유지 확인** | powercfg 적용 후 장시간 방치해도 화면이 꺼지지 않는지 확인, 화면 켜진 상태에서 캡처 정상 동작 확인 | 30분 |
| 모바일 접속 확인 | 브라우저에서 Bridge health 응답 확인 (LAN + LTE) | 30분 |

> Phase 0 조기 검증 2건이 실패하면 Phase 1 착수 전 대응 방안을 재검토한다.
>
> **디스플레이 유지 원리**: Bridge는 `powercfg /change monitor-timeout-ac 0`으로 OS가 디스플레이를 자동으로 끄는 것을 막는다. 따라서 Bridge 실행 중에는 화면이 항상 켜진 상태이고, Cursor(Electron)도 정상적으로 렌더링을 유지한다. Phase 0에서는 이 설정이 실제로 적용되어 장시간 방치 후에도 화면이 꺼지지 않는지를 확인한다.
> 단, 물리 버튼으로 모니터를 직접 끄거나 PC 시스템이 절전 상태(S3/S4)에 진입하는 경우는 이 설정으로 막을 수 없으며, 설계 범위 밖이다.

## 6. 보안

| 위협 | 대응 |
|------|------|
| 무단 외부 접속 | Tailscale ACL — 내 기기만 접근 |
| Bridge 포트 노출 | localhost 바인딩, Tailscale 통해서만 접근 |
| 인증 | pairing code → JWT (30일) |
| 모바일 분실 | Tailscale 관리 콘솔에서 해당 기기 즉시 제거 → 토큰 유효기간과 무관하게 Bridge 접근 즉시 차단 |

> Tailscale 연동 계정(Google 등)에 **2단계 인증** 필수.
>
> **JWT 특성**: JWT는 stateless 방식이라 서버 측에서 강제 만료(revoke)가 불가능하다. 모바일 분실 시 Tailscale에서 기기를 제거하면 Bridge 네트워크 자체에 도달할 수 없게 되므로, 잔여 JWT 유효기간은 실질적 위협이 되지 않는다.

## 11. 모바일 클라이언트 — iPhone 앱 vs Safari/Chrome (논의 필요)

> **상태:** Phase 1 착수 전 **방향 논의**. Phase 1은 **웹(Safari) 우선**으로 진행하고, Phase 1 실사용 후 §11.5 기준으로 확정한다.

### 11.1 선택지 요약

| 방식 | 설명 | Phase 1 |
|------|------|---------|
| **A. Safari / Chrome (웹)** | Bridge가 제공하는 HTML 페이지. 홈 화면 추가 가능(PWA) | **권장·1차 구현** |
| **B. WKWebView 쉘** | Swift로 만든 얇은 iOS 앱, 내부는 동일 웹 UI 로드 | Phase 3 후보 |
| **C. 네이티브 SwiftUI** | MJPEG·채팅 UI·인증 전부 iOS 네이티브 | Phase 3+, 비용 큼 |

**Chrome on iOS 참고:** Apple 정책상 **Chrome·Firefox도 WebKit 엔진**을 씀 → Safari와 렌더링·제약이 **거의 동일**. “Chrome vs Safari”보다 **“웹 vs 네이티브”** 가 실질적 선택이다.

### 11.2 B모드(Phase 1) 기능별 적합도

| 기능 | Safari/Chrome 웹 | 네이티브 iOS 앱 |
|------|------------------|-----------------|
| MJPEG `<img src="/stream">` | ✅ iOS Safari 지원 | ✅ AVFoundation 등 |
| Tailscale 접속 | ✅ **OS VPN** (앱 불필요) | ✅ 동일 |
| pairing + JWT | ✅ localStorage | ✅ Keychain (보안↑) |
| 텍스트 입력·전송 | ✅ | ✅ |
| 홈 화면 바로가기 | ✅ “홈에 추가” | ✅ 앱 아이콘 |
| **백그라운드 MJPEG** | ❌ 탭 suspend 시 끊김 | ⚠️ 제한적 (iOS 정책) |
| **푸시 알림** (Agent 응답) | ⚠️ Web Push (iOS 16.4+, PWA 설치 필요) | ✅ APNs — 구현 용이 |
| App Store / TestFlight | 불필요 (URL 접속) | 필요 (배포·갱신 부담) |
| 개발·유지 공수 | **낮음** (1 codebase) | **높음** (Swift + Bridge) |

### 11.3 Planner 의견 — Phase 1 효율성

**Phase 1에는 Safari(웹)가 더 효율적**이다.

1. **B모드 핵심(MJPEG + 입력 폼)** 은 순수 웹으로 충분하다.
2. Tailscale은 **iOS 시스템 VPN** 이라 브라우저·앱 차이가 없다.
3. Chrome iOS는 Safari와 엔진이 같아 **이중 테스트 가치가 낮다** — Safari만 1차 타겟으로 한다.
4. 개인용·비상용 목적이면 **App Store 없이 URL + 홈 화면 추가**로 충분하다.
5. Bridge·캡처·클립보드(PC측)가 병목이지, 모바일 UI 프레임워크가 아니다.

**네이티브 앱이 유리해지는 시점 (Phase 3 검토 트리거)**

- Agent **푸시 알림**이 필수일 때
- 앱을 **잠금 후 Face ID** 로 보호하고 싶을 때
- Safari에서 MJPEG **가독성·성능**이 실사용에 부족할 때
- **Win+L 잠금 대응**으로 CLI/API 경로를 붙일 때 (UI와 무관하게 백엔드 변경)

### 11.4 잠금 화면과 클라이언트 형태 (Phase 3 연계)

| PC 상태 | Safari/Chrome | 네이티브 앱 |
|---------|---------------|-------------|
| 모니터 ON, Cursor 전면 (Phase 1) | ✅ B모드 | ✅ B모드 |
| Win+L 잠금 (Phase 3 목표) | ❌ B모드 불가 | ❌ B모드 불가 — **동일** |

→ **잠금 대응은 iPhone 앱 vs 브라우저 문제가 아니라 PC측 아키텍처 문제**다. Phase 3에서 GUI(B모드) 대신 CLI/API·RDP 등을 검토한다.

### 11.5 결정 기준 (Phase 1 완료 후 체크)

Phase 1 E2E 후 아래를 채워 **클라이언트 방향 확정**한다.

- [ ] iPhone Safari에서 MJPEG 가독성·지연이 **실사용 acceptable** 한가?
- [ ] 백그라운드에서 영상 끊김이 **허용 가능**한가? (대화 중만 전면 사용 vs 상시 모니터링)

## 부록 — 용어

| 용어 | 설명 |
|------|------|
| B모드 | 화면 스트리밍 방식. Cursor 패널을 영상으로 모바일에 중계 |
| A모드 | Extension 방식. 대화 내용을 텍스트로 추출해 채팅 UI로 표시 |
| Bridge | PC에서 상주하며 모바일과 Cursor를 중계하는 서버 |
| Tailscale | 설정이 거의 필요 없는 개인용 VPN. 집 밖에서도 PC에 안전하게 연결 |
| MJPEG | 화면을 연속 이미지로 스트리밍하는 방식. WebRTC보다 단순하고 구현이 쉬움 |
| 클립보드 주입 | Bridge가 PC 클립보드에 텍스트를 넣고 Cursor 입력창에 붙여넣기하는 방식 |
| pairing code | 처음 연결 시 한 번만 입력하는 6자리 인증 코드 |
| JWT | 이후 요청마다 자동으로 사용되는 전자 출입증 (유효기간 30일) |
| PWA | Safari「홈 화면에 추가」로 앱처럼 실행하는 웹 앱 |
| WKWebView | iOS 네이티브 앱 내부에 웹 UI를 넣는 방식 (Phase 3 후보) |
